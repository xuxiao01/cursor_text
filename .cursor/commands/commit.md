请基于当前暂存区改动（staged changes）执行一次“审查 + 提交”的流程。

第 1 步：先做首轮代码审查
请先只基于当前暂存区改动进行 review，不要泛泛分析整个项目。

重点检查：
1. 边界条件是否完整
2. 是否存在空值、undefined、null 或异常情况未处理
3. 是否存在明显的类型问题、隐式问题或不必要的 any
4. 是否引入重复逻辑，是否有可抽离的公共逻辑
5. 命名是否清晰、结构是否易维护
6. 是否可能影响原有功能或引入副作用
7. 是否缺少 loading、错误处理或状态恢复
8. 是否存在明显的可读性或可维护性问题

请先输出：
## 1. Review 摘要
- 简洁总结本次暂存区改动在做什么

## 2. Review 问题
- 按条列出问题
- 每条问题标注风险等级：高 / 中 / 低
- 说明对应原因

## 3. Review 结论
- 如果存在高风险问题，请明确写出“建议修复后再提交”
- 如果没有明显阻塞问题，请明确写出“可继续生成 commit”

第 2 步：生成 commit message
如果没有明显阻塞问题，请基于当前暂存区改动生成一条详细的 git commit message。

要求：
1. 尽量准确概括本次改动
2. 推荐使用 Conventional Commits 风格，例如 feat / fix / refactor / chore / docs
3. commit message 需要包含：
   - 标题
   - 必要时补充 2~4 条变更说明

输出格式：
## Commit Message
<在这里给出完整 commit message>

第 3 步：执行提交
在给出 commit message 后，请询问我是否确认提交。
只有在我明确确认后，才执行 git commit。
不要自动修改代码。
